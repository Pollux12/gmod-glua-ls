## Overview

Garry's Mod uses a "scripted class" pattern where you use a global table (`ENT`, `SWEP`, etc.) in a specific file location and GMod registers it as a class. GLuaLS detects these classes automatically and provides:

- Automatic class registration including inheritance
- Automatic synthesis of `NetworkVar` getter/setter pairs
- Automatic synthesis of `AccessorFunc` getter/setter pairs
- Better completions and type checking for scripted class code

---

## Automatic detection

GLuaLS detects scripted classes by their **file path**. Any file under a recognized directory is treated as defining the appropriate table. For example:

| Directory pattern | Class table | Base class |
|---|---|---|
| `lua/entities/**/` | `ENT` | `Entity` |
| `lua/weapons/**/` | `SWEP` | `Weapon` |
| `lua/effects/**/` | `EFFECT` | (none) |

The detected class table is automatically typed with the base class's methods and fields, so you get completions and type-checking without any annotation.

This table can be adjusted in settings.

---


## AccessorFunc synthesis

`AccessorFunc(table, member, name, ...)` generates getter and setter methods. GLuaLS detects these calls and synthesizes the methods:

```lua
AccessorFunc(ENT, "m_bOrient", "Orient", FORCE_BOOL)

-- GLuaLS synthesizes:
-- ENT:GetOrient() -> boolean
-- ENT:SetOrient(value: boolean) -> nil
```

This can be done via the annotation `---@accessorfunc` for custom getters/setters, see [@accessorfunc](/annotations/accessorfunc) for more details.

---

## NetworkVar synthesis

`ENT:NetworkVar(type, slot, name, ...)` generates entity accessors. GLuaLS detects these calls and synthesizes matching methods:

```lua
function ENT:SetupDataTables()
    self:NetworkVar("Int", 0, "Health")
    self:NetworkVar("Bool", 1, "Active")
    self:NetworkVar("String", 2, "Name")
end

-- GLuaLS synthesizes:
-- ENT:GetHealth() -> number
-- ENT:SetHealth(value: number) -> nil
-- ENT:GetActive() -> boolean
-- ENT:SetActive(value: boolean) -> nil
-- ENT:GetName() -> string
-- ENT:SetName(value: string) -> nil
```

This gives you completions and type checking for generated accessors without manual annotations.

### Supported NetworkVar types

| NetworkVar type | Getter/setter type |
|---|---|
| `"Int"` | `integer` |
| `"UInt"` | `integer` |
| `"Float"` | `number` |
| `"Double"` | `number` |
| `"Bool"` | `boolean` |
| `"String"` | `string` |
| `"Entity"` | `Entity` |
| `"Vector"` | `Vector` |
| `"Angle"` | `Angle` |
| `"Color"` | `Color` |

## Base class inheritance

Scripted classes in GMod inherit from base classes. GLuaLS understands this chain and provides completions for inherited methods. The scripted class table also gets the standard base type for that scope (`Entity`, `Weapon`, `Tool`, and so on). `DEFINE_BASECLASS(...)` and values like `ENT.Base` can refine that further.

```lua
function ENT:Use(ply)
    self:EmitSound("...")    -- inherited from Entity / base class chain ✅
    ply:ChatPrint("Hello")   -- Player method ✅
end
```

---

## Hook overrides in scripted classes

GLuaLS understands that functions defined in a scripted class table override the engine hooks for that class:

```lua
function ENT:Think()    -- override of ENTITY:Think hook
function ENT:Draw()     -- override of ENTITY:Draw hook
function ENT:OnRemove() -- override of ENTITY:OnRemove hook
```

Completions suggest the correct signature for each override.

## Class Explorer integration

Detected scripted classes are listed in the VS Code **Class Explorer**. You can browse groups (entities, weapons, effects, and custom scopes) and jump to class definitions quickly.

Class grouping and labels come from `gmod.scriptedClassScopes.include`.

---

## Scaffolding integration

Scripted class scopes also drive class scaffolding. Each scope can define scaffold output files via `scaffold.files` and a `rootDir` in `gmod.scriptedClassScopes.include`.

This lets you create new classes with your own template layout while keeping class analysis, explorer grouping, and generated files in sync.

See [GMod settings](/configuration/gmod#scripted-classes) for all supported fields.

### Creating a new class

1. Open the **Class Explorer** in the sidebar.
2. Find the group you want, such as **Entities**.
3. Click the **+** button next to that group.
4. Enter the class name.
5. The VS Code extension creates the files and opens the main file.

### Built-in templates

The VS Code extension ships with built-in templates for the standard scripted class types:

| Class type | Generated files |
|---|---|
| **Entity** | `shared.lua`, `init.lua`, `cl_init.lua` |
| **Weapon (SWEP)** | `shared.lua` |
| **Effect** | `your_effect_name.lua` |
| **Scripted Tool (STOOL)** | `your_tool_name.lua` |

## Custom class scopes

If you use a non-standard directory structure or custom class types, configure GLuaLS to recognize them:

<ConfigTip />

```json .gluarc.json
{
  "gmod": {
    "scriptedClassScopes": {
      "include": [
        {
          "id": "my_panels",
          "label": "Panels",
          "classGlobal": "PANEL",
          "path": ["panels"],
          "include": ["lua/panels/**"],
          "rootDir": "lua/panels"
        }
      ]
    }
  }
}
```

### Fields

| Field | Description |
|---|---|
| `id` | Unique identifier for this class scope |
| `label` | Display name in the Class Explorer sidebar |
| `classGlobal` | The global table name used in scripts (e.g., `ENT`, `SWEP`, `PANEL`) |
| `path` | Path segments used for display grouping in the Class Explorer |
| `include` | Glob patterns matching files that belong to this class type |
| `rootDir` | Root directory for scaffolded files |
| `icon` | VS Code icon ID shown in the Class Explorer (e.g., `symbol-class`) |

---



## Disabling scripted class inference

To disable a built-in scripted class scope, add an override with the same `id` and set `"disabled": true`.

```json .gluarc.json
{
  "gmod": {
    "scriptedClassScopes": {
      "include": [
        {
          "id": "effects",
          "disabled": true
        }
      ]
    }
  }
}
```
