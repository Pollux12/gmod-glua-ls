## Overview

GMod's hook system is central to almost every addon and gamemode. GLuaLS understands `hook.Add`, `hook.Remove`, `hook.Run`, `hook.Call`, and `GM:` definitions — providing completions, type inference, and validation for all of them.

---

## Hook name validation

Unrecognized hook names in `hook.Add` and `hook.Remove` trigger a diagnostic:

```lua
hook.Add("PlayerSpan", "MyAddon", function(ply) -- ❌ did you mean PlayerSpawn?
    -- ...
end)
```

All base gmod hooks are recognised via the base gmod hook definitions. Custom hooks defined elsewhere in your workspace via `hook.Run("MyCustomHook", ...)` are also tracked.

<Info>
For custom hooks, while autocomplete will work as long as it has been used once, it will lack any "Goto" definition, parameter type inference, or other advanced hook features unless a `GM:` function has been defined for that hook.
</Info>

---

## Parameter type inference

When GLuaLS knows the hook name, it automatically types the callback parameters:

```lua
hook.Add("PlayerSpawn", "MyAddon", function(ply)
    --                                       ^^^
    --                                       typed as Player automatically
    local hp = ply:Health() -- ✅ typed, Player method
end)

hook.Add("EntityTakeDamage", "MyDamageHook", function(target, dmg)
    -- target: Entity
    -- dmg: CTakeDamageInfo
    -- All typed from the GMod wiki annotations
end)
```

---

## Callback snippet completion

When you type `hook.Add("HookName",` and trigger completion on the callback argument, GLuaLS inserts a fully typed function snippet:

```lua
-- Accepting completion for hook.Add("PlayerSpawn") inserts:
hook.Add("PlayerSpawn", "MyAddon", function(ply)
    -- YOUR CODE
end)
```

The snippet includes the correct number of parameters with their names.

---

## GM: hook definitions

Files in a gamemode context (e.g., `gamemodes/*/gamemode/`) define `GM` as the gamemode table. GLuaLS recognizes `function GM:HookName()` as a hook override:

```lua
function GM:PlayerSpawn(ply)
    -- ply is typed as Player ✅
    self.BaseClass.PlayerSpawn(self, ply)
end
```

A `GM:` hook definition is required for advanced features to work on custom hooks.

---

## hook.Run and hook.Call completions

When typing the first argument to `hook.Run` or `hook.Call`, GLuaLS autocompletes from known hook names:

```lua
hook.Run("PlayerSpawn", ply)  -- "PlayerSpawn" autocompletes
```

---

## Custom hooks

Hooks created by your code (or your library) are also tracked:

```lua
-- Somewhere in your code:
hook.Run("MyAddon.OnPlayerAction", ply, action)

-- Elsewhere:
hook.Add("MyAddon.OnPlayerAction", "handler", function(ply, action)
    -- ply and action types are inferred from call sites
end)
```

<Info>
Advanced features such as parameter type inference, hover descriptions, function validation, usages and view definitions will only work for custom hooks if a `GM:` definition has been created for that hook.
</Info>

---

## VSCode integration

Hook intelligence works automatically when you use `hook.Add`, `hook.Run`, or define `GM:` methods. Configure validation behavior in the [settings panel](/configuration/overview#settings-panel-recommended) — see [GMod settings](/configuration/gmod#hooks) for the full reference.

## Configuration

Hook validation and mappings are now controlled through the `gmod` section in `.gluarc.json`, in particular `gmod.hookMappings` and the global diagnostics settings.

| Setting | Default | Description |
|---|---|---|
| `gmod.hookMappings.methodToHook` | `{}` | Map method names to hook names (e.g. `"Think": "Think"`) |
| `gmod.hookMappings.emitterToHook` | `{}` | Map custom emitter functions to hook names |
| `gmod.hookMappings.methodPrefixes` | `[]` | Custom method prefixes for inferred `GM:` hook methods |
| `diagnostics.disable` | `[]` | Disable individual diagnostics (e.g., `"gmod-invalid-hook-name"`) |
