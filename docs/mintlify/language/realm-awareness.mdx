## Overview

Garry's Mod Lua runs in two **realms**: clientside (`CLIENT`) and serverside (`SERVER`), with code accessible on both being referred to as "shared". Many GMod API functions are only available in one realm (e.g., `LocalPlayer()` is client-only), with cross-realm mistakes being a common source of bugs for GMod developers. GLuaLS is fully realm aware to help resolve this.

GLuaLS automatically detects which realm each file runs in and issues diagnostics when you call realm-restricted functions from the wrong realm.

---

## Realm detection

GLuaLS contains an advanced realm detection system that uses multiple detection methods to determine the realm of each file or even within each file. This system is designed to "just work" and requires little to no configuration for almost all projects.

GLuaLS infers realm through a **5-level priority** system, from highest to lowest:

### 1. `@realm` annotation (highest priority)

<Warning>
**It is not recommended to add file level realm overrides where the realm is already correctly inferred**, as this can mask potential cross-realm issues. Use function-level overrides instead where possible.
</Warning>

Put `---@realm` at the top of a file or on a function:


```lua
---@realm client
-- Everything in this file is clientside
```

See the [`@realm` annotation reference](/annotations/realm) for the full syntax. Annotations override all other systems, with function level annotations taking precedence over file-level annotations.

### 2. Filename prefix detection

Files beginning with `cl_`, `sv_`, or `sh_` are assigned their realm automatically:

| Prefix | Realm |
|---|---|
| `cl_` | Client |
| `sv_` | Server |
| `sh_` | Shared |

### 3. Directory-based detection

Files in well-known directories are assigned a realm:

| Path pattern | Realm |
|---|---|
| `lua/autorun/client/` | Client |
| `lua/autorun/server/` | Server |
| `lua/autorun/` | Shared |
| `lua/client/` or `lua/vgui/` | Client |
| Other recognized GMod paths | Shared or unresolved |

### 4. Block-level narrowing

Within shared files, GLuaLS narrows realm within `if CLIENT` / `if SERVER` blocks:

```lua
if CLIENT then
    -- GLuaLS treats this block as client-only
    surface.DrawRect(0, 0, 100, 100) -- OK here
end

if SERVER then
    -- This block is server-only
    ply:SetHealth(100) -- OK here
end
```

### 5. Heuristic inference (lowest priority)

If none of the above apply, GLuaLS may infer realm from the code within a file (e.g., calling a server-only function implies server realm). This is a best-effort heuristic and may not always be accurate, but it can catch cases where realm is not explicitly defined.

## Realm diagnostics

| Code | Description |
|---|---|
| `gmod-realm-mismatch` | Confident realm mismatch based on annotations |
| `gmod-realm-mismatch-heuristic` | Probable realm mismatch based on inference |
| `gmod-unknown-realm` | Realm could not be determined |

---

## Disabling realm checks

<Warning>
It is not recommended to disable the realm checks, as this will significantly degrade the quality of other features which depend on it. It is instead recommended to suppress specific diagnostics via the settings menu - heuristic diagnostics have a separate setting for exactly this reason.
</Warning>

Disable realm diagnostics workspace-wide:

<ConfigTip />

```json .gluarc.json
{
  "diagnostics": {
    "disable": ["gmod-realm-mismatch", "gmod-realm-mismatch-heuristic", "gmod-unknown-realm"]
  }
}
```

Or suppress for a specific file:

```lua
---@realm shared
-- This file intentionally has mixed realm code
```

---

## Per-function realm overrides

Override or set realm detection for an individual function:

```lua
---@realm server
function MyAddon.ServerAction()
    -- This function is always treated as server-side,
    -- even if the surrounding file is shared
end
```

It is recommended to use function-level overrides where possible, since this helps speed up performance of the language server, as it does not have to perform expensive inference when annotations are present.

## Realm in hover and autocomplete

Every hover tooltip for a GMod API function includes a realm badge showing where it runs:

- **Client** badge — function is clientside only
- **Server** badge — function is serverside only
- **Shared** badge — function is available on both realms

The badge appears in completions as well, so you can check a function's realm before inserting it.

The realm is used to filter autocomplete suggestions to reduce noise and prevent invalid code from being suggested. For example, if you're editing a clientside file, server-only functions will be hidden from autocomplete and vice versa. If it seems like some functions are missing from autocomplete, check the realm!
