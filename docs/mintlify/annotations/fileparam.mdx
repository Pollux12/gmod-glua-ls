## Overview

`@fileparam` sets a default type for any parameter with a matching name across all functions in the current file. Functions that don't have an explicit `@param` for that name fall back to the file-level default.

---

## Syntax

```lua
---@fileparam paramName type
```

---

## Usage

```lua
---@fileparam ply Player
---@fileparam vehicle Vehicle

-- No @param needed for these functions — they inherit from @fileparam
local function enter(vehicle, ply)
    local driver = vehicle:GetDriver()  -- vehicle is typed as Vehicle ✅
    if IsValid(driver) then
        driver:ChatPrint("Welcome!")
    end
    ply:EnterVehicle(vehicle)           -- ply is typed as Player ✅
end

local function exit(vehicle, ply)
    ply:ExitVehicle()  -- ✅
end
```

---

## Precedence

If GLuaLS cannot already infer the parameter type from the function signature or owning type, it falls back in this order:

1. **Explicit `@param`** annotation on the function itself
2. **`@fileparam`** for the current file
3. **`gmod.fileParamDefaults`** in `.gluarc.json` — workspace-wide defaults

---

## Override with @param

If one function in the file needs a different type for the same parameter name, add an explicit `@param`:

```lua
---@fileparam vehicle Vehicle

local function enter(vehicle, ply)
    -- vehicle: Vehicle (from @fileparam)
end

---@param vehicle Entity  -- overrides @fileparam for this function only
local function inspectGeneric(vehicle)
    -- vehicle: Entity
end
```

## Workspace defaults

Configure defaults that apply to all files in the workspace:

<ConfigTip />
```json .gluarc.json
{
  "gmod": {
    "fileParamDefaults": {
      "ply": "Player",
      "target": "Entity",
      "dmginfo": "CTakeDamageInfo"
    }
  }
}
```

---

## Common patterns

In GMod addon development, certain parameter names are almost always the same type. Using `@fileparam` or workspace defaults cuts down on repeated `@param` annotations:

```lua
---@fileparam ply Player
---@fileparam attacker Player
---@fileparam victim Entity
```
