## VSCode extension (recommended)

The VS Code extension is the recommended way to use GLuaLS. It handles setup automatically:
- downloads the language server binary
- downloads GMod wiki annotations
- installs debugger pieces
- keeps these updated

<Steps>
  <Step title="Install from the marketplace">
    Search for **GLua Language Server** in the VS Code Extensions panel (`Ctrl+Shift+X`) or install directly from the [Visual Studio Marketplace](https://marketplace.visualstudio.com/items?itemName=Pollux.gmod-glua-ls).
  </Step>
  <Step title="Open your project">
    Open your addon or gamemode folder in VS Code. The extension expects each workspace folder to be a single addon or gamemode root (the folder that directly contains `lua/`, `addon.json`, etc.).

    For working on multiple addons at once, use a [multi-root workspace](https://code.visualstudio.com/docs/editing/workspaces/multi-root-workspaces).
  </Step>
  <Step title="Let the extension initialize">
    On first launch, the extension automatically:
    - Downloads the language server binary
    - Downloads the latest GMod wiki annotations
    - Sets up the `.gluarc.json` configuration file if needed

    A notification in the bottom-right corner confirms when initialization is complete.
  </Step>
</Steps>

<Warning>
Do not install other Lua language server extensions alongside GLuaLS. Extensions such as EmmyLua, LuaLS, and GLua Enhanced will conflict. Create a dedicated VS Code profile for GMod development if you need other Lua tools for unrelated projects.
</Warning>

## Workspace structure

GLuaLS works best when each workspace folder represents a single addon or gamemode. The extension auto-detects the structure and configures itself accordingly.

<Tip>
If you are working on a gamemode with multiple addons, add each addon as its own folder in a multi-root workspace instead of opening the whole `garrysmod/` folder.
</Tip>


**Expected structure:**

<Tree>
  <Tree.Folder name="garrysmod" openable="false" defaultOpen>
    <Tree.Folder name="addons" openable="false" defaultOpen>
      <Tree.Folder name="my-addon <- Workspace Root" defaultOpen>
        <Tree.File name="lua" />
        <Tree.File name="addon.json" />
        <Tree.File name="..." />
      </Tree.Folder>
    </Tree.Folder>
    <Tree.Folder name="gamemodes" openable="false" defaultOpen>
      <Tree.Folder name="my-gamemode <- Workspace Root" defaultOpen>
        <Tree.File name="gamemode" />
        <Tree.File name="..." />
      </Tree.Folder>
    </Tree.Folder>
  </Tree.Folder>
</Tree>

Each VS Code workspace root should match one folder shown above. Use one root for the gamemode and one root per addon. For multiple addons, use a multi-root workspace instead of opening the whole `garrysmod/` folder.

<Warning>
All folders within the multi-root workspace must be from the same game directory. Do not mix some folders from client and some from SRCDS.
</Warning>

For example, if you are working on a gamemode with two addons, you would open each individually into VSCode as separate workspaces.

This would be a multi-root workspace with three folders added:
- `garrysmod/addons/my-addon-1/`
- `garrysmod/addons/my-addon-2/`
- `garrysmod/gamemodes/my-gamemode/`

With this setup, your VSCode explorer should look similar to this:

<Tree>
  <Tree.Folder name="my-addon-1" defaultOpen>
    <Tree.File name="lua" />
    <Tree.File name="addon.json" />
    <Tree.File name="..." />
  </Tree.Folder>
  <Tree.Folder name="my-addon-2" defaultOpen>
    <Tree.File name="lua" />
    <Tree.File name="addon.json" />
    <Tree.File name="..." />
  </Tree.Folder>
  <Tree.Folder name="my-gamemode" defaultOpen>
    <Tree.File name="gamemode" />
    <Tree.File name="..." />
  </Tree.Folder>
</Tree>

<Tip>
If you're using a multi-root workspace, you only need to set the debugger up for one of the folders. The same configuration will be valid for anything within the same Garry's Mod directory.
</Tip>

## Verify everything is working

Open any `.lua` file in your project. You should immediately see:

- **Syntax highlighting** with semantic token coloring
- **Autocomplete** as you type (press `Ctrl+Space` to trigger manually)
- **Hover documentation** when you hover over GMod API functions
- **Realm badges** next to functions showing their client/server/shared status
- **Diagnostics** as squiggly underlines for potential issues

Workspace indexing usually finishes in a few seconds after extension startup, even on larger projects.

<Tip>
If you are not seeing any diagnostics but expected to, check the Garry's Mod Language Server section in the output panel for any errors.
</Tip>

## Other editors

GLuaLS works with any LSP-compatible editor, but many advanced features (debugger, Class Explorer, settings UI, and more) are VS Code extension features.

<Note>
Zed support is planned. It will have fewer features than the VSCode extension due to Zed's more limited extension API. Other editors are not currently planned.
</Note>

### Install the binary

```bash
cargo install glua_ls glua_check
```

Point your LSP client to the `glua_ls` binary. Most editors accept a `cmd` or `command` setting in their LSP configuration.

### Build from source

```bash
git clone https://github.com/Pollux12/gmod-glua-ls.git
cd gmod-glua-ls
cargo build --release
```

The binary is at `target/release/glua_ls` (or `glua_ls.exe` on Windows).
